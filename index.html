第十一章 关联容器

关联容器支持高效的关键字查找和访问。两个主要的关联容器(associative-container)类型是map和set。map中元素是关键字-值对(键值对key-value)：关键字起到索引作用， 值表示与所以关联的数据。set中每个元素只包含一个关键字，set支持高效的关键字查询-给定关键字是否在set中。

标准库提供8个关联容器，map和multimap定义在头文件map中；set和multiset定义在头文件set中；无序容器定义在头文件unordered_map和unordered_set中。

关联容器 说明
map 保存键值对
set 关键字即值，只保存关键字
multimap 关键字可以重复出现的map
multiset 关键字可以重复出现的set
unordered_map 用哈希函数组织的map
unordered_set 用哈希函数组织的set
unordered_multimap 关键字可以重复出现的用哈希组织的map
unordered_multiset 关键字可以重复出现的用哈希组织的set

第十章 泛型算法

标准库定义了一组泛型算法(generic algorithm)，称为“算法”因为他们实现了一些经典算法的公共接口，如搜索和排序；称为“泛型”因为可以用于不同类型的元素和多种容器类型。

10.1 概述

大多数算法定义在头文件algorithm中，还在numeric头文件中定义了一组数值泛型算法

一般情况下，算法不直接操作容器，而是遍历由两个迭代器指定的一个范围。

如算法find

find前两个参数指定范围，第三个参数表示要寻找的元素。如存在，返回指向第一个等于给定值的元素的迭代器；不存在返回第二个参数，即范围结束的迭代器。

算法依赖于迭代器，永远不会执行容器的操作。

10.2 初识泛型算法

除少数外，标准库算法都对一个范围内的元素进行操作，这个范围称为“输入范围”。这个范围由两个参数表示，分别指向要处理的第一个元素，和要处理的最后一个元素之后的位置。

10.2.1 只读算法

对于只读算法，通常最好用cbegin()和cend()，除非有特殊需求。

一些算法只读取输入范围内的值，不改变元素。如：find、count、accumulate(定义在numeric头文件)

equal用来确定两个序列的值是否相同(依次比较对应位置上的值)。接受三个参：前两个表示范围，第三个表示第二个序列的首元素(第二个序列至少和第一个一样长)

第九章 顺序容器

一个容器就是一些特定类型对象的集合。顺序容器(sequential container)为程序员提供了控制元素存储和访问顺序的能力。这种顺序不依赖于元素的值，而是与元素加入容器时的位置对应。

9.1 顺序容器概述

下表列出了标准库中的顺序容器，使用顺序容器都提供了快速顺序访问的元素的能力。但是，这些容器在一些方面有不同的性能折中

  • 向容器添加或从容器中删除元素的代价
  • 非顺序访问容器中元素的代价
容器类型 说明
vector 可变大小数组，支持快速随机访问。在尾部之外的位置插入或者删除元素可能很慢
deque 双端队列。支持快速随机访问，在头尾位置插入/删除元素速度很快
list 双向链表。只支持双向顺序访问，在list中任何位置插入/删除操作速度都很快
forward_list 单项链表。只支持单向顺序访问，在链表任何位置插入/删除操作速度都很快
array 固定大小数组。支持快速随机访问，不能添加或者删除元素
string 与vector类似的容器，专门用来保存字符。随机访问快，在尾部插入/删除速度快

确定使用哪种顺序容器

  • 除非你有很好的利用选择其他容器，否则应该使用vector
  • 如果你的程序有很多小的元素，且空间的额外开销很重要，则不要使用list或者forward_list
  • 如果程序要求随机访问元素，应该使用vector或deque
  • 如果程序要求在容器的中间插入或删除元素，应该使用list或forward_list
  • 如果程序需要在头尾位置插入删除元素，但不会在中间位置插入删除元素，使用deque
  • 如果程序只有在读取输入时才需要在容器中间位置插入元素，随后需要随机访问元素，则
    • 首先，确定是否真的要在容器中间位置添加元素。当处理输入数据时，通常可以很容易向vector追加数据，然后调用标准库的sort函数来重排容器中元素，从而避免在中间位置添加元素
    • 如果必须在中间位置插入，考虑在输入阶段使用list，一旦输入完成，将list中内容拷贝到一个vector中

一般来说，应用中站主导地位的操作(访问操作还是插入/删除)决定了容器类型的选择。

如果不确定使用哪种容器，可以在程序中只使用vector和list的公共操作：迭代器，不用下标操作，避免随机访问。这样在必要时选择使用vector和list都方便。

9.2 容器库概览

容器类型上的操作形成了一种层次：

  • 某些操作时所有容器类型都提供的
  • 另外一些操作仅针对顺序容器、关联容器或无序容器
  • 还有一些操作只适用于一小部分容器

一般来说，每个容器都定义在一个头文件中，文件名和类型名相同

8.1.3 管理输出缓冲

执行这行代码，文本串可能立即打印，也有可能被保存到缓冲区，随后打印。

缓冲机制使得操作系统可以将程序的多个输出操作组合成单一的系统级写操作，带来性能提升。

导致缓冲刷新(即：数据真正写到输出设备或者文件)的原有有很多：

  • 程序正常结束，作为main函数的return操作的一部分，缓冲刷新被执行
  • 缓冲区满，需要刷新缓冲，使得新的数据可以写入缓冲区
  • 使用操纵符如endl来显示刷新缓冲区
  • 每个输出操作后，使用操纵符unitbuf设置流内部状态，清空缓冲区。默认情况下，对cerr是设置unitbuf的，因此写到cerr的内容都是立即刷新的
  • 一个输出流可能被关联到另外一个流。这种情况下，当读写被关联的流时，关联到的流的缓冲区会被刷新。如默认情况下，cin和cerr都关联到cout，因此读cin或者写cerr都会导致cout的缓冲区被刷新

刷新输出缓冲区

endl、flush、ends

unitbuf操纵符

unitbuf告诉流在接下来的每次写操作之后都执行一次flush操纵；nounitbuf操纵符重置流，使其恢复正常的系统管理的缓冲区刷新机制

警告：若程序崩溃，输出缓冲区不会被刷新

当一个程序崩溃后，它所输出的数据很可能停留在输出缓冲区等待打印。调试这样的程序时，需要确认你认为已经输出的数据确实已经刷新了，否则会浪费大量时间追踪代码为什么没有执行，但实际上已经执行，只是缓冲区没有刷新，数据数据未打印。

8.2.2 文件模式

文件模式 说明
in 读方式打开
out 写方式打开
app 每次写操作前均定位到文件末尾
ate 打开文件后立即定位到文件末尾

指定文件模式有如下限制：

  • 只可以对ofstream或fstream对象设定out模式
  • 只可以对ifstream或fstream对象设定in模式
  • 只有当out也被设定时才可以设定trunc模式

每个文件流都定义了一个默认的文件模式：ifstream关联的文件默认以in模式打开；ofstream关联的文件默认以out模式打开；fstream关联的文件默认以in和out模式打开

类基础

面向过程与面向对象

面向过程的程序中函数是构成程序的基本单位，每实现一个新功能就新写一个函数，之后调用就可以。

面向对象编程就不是为了实现功能而写个函数。以狗为例，狗有很多属性，如品种、体型大小、毛发、颜色，也会有很多动作，跳、吠，描述一个对象时应该包括这些要素。

任何一个对象都应该有属性和行为这两个要素，一个对象也是由多个属性和多个行为构成的。

在面向对象的编程中，对象由数据和函数两个部分构成，数据就是属性，函数就是行为。

类的声明

类是对象的类型，用来表示一类具有共同属性和行为的类型，对象是类的实例，真真实实的变量，占用存储空间，而类不占空间。
